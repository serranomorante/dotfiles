# Dependency Pin Updates

Pinned package versions in Ansible are tracked by Renovate where the upstream version source is stable enough to model. Renovate provides the dependency metadata, while `dotfiles-renovate-apply` is the local apply layer: it reads the root `renovate.json`, creates a temporary Git worktree on a `renovate-pins/<timestamp>` branch, resolves supported upstream versions, edits the matched pins, and commits the result locally for review. Ansible remains responsible for installing accepted pins and `update-diff-capture` remains responsible for storing install-time diffs.

The root `renovate.json` uses only `custom.regex` managers so repository-specific Ansible variables and package lists can be updated without adding inline Renovate comments to role YAML. Keep dependency metadata in the Renovate config, not beside the variables it updates.

Renovate is used locally with `--platform=local`; it must not be run as a GitHub PR bot for this repository. Local mode does not create local commits, branches, or PRs: Renovate's local platform has no-op `commitAndPush` and `createPr` implementations. Dashboard approval is disabled because local mode has no GitHub issue dashboard to approve updates from. The Ansible-managed `renovate` wrapper changes to `~/dotfiles` before running so local mode sees the root `renovate.json` and repository-relative manager patterns, runs with the dedicated Node 24 dependency-update runtime, stores npm modules under a Node-versioned prefix so native modules are not reused across Node ABIs, and injects `--onboarding=false --require-config=required` so local runs use the tracked root config instead of trying to create onboarding state. The Ansible-managed `dotfiles-renovate-apply` helper is the command to use when a local branch and commit should be generated automatically; lazygit exposes it as a global custom command. GitHub authentication for Renovate goes through `utilities/bin/dotfiles-github-token`, which reads non-empty token environment variables first and then the KWallet entry at wallet `kdewallet`, folder `dev-tools`, entry `github-temporal-access-token`; an absent or empty token is normal and means requests run anonymously.

Renovate currently covers managed npm package lists, selected Node and Python runtime lanes, the explicit Piper PyPI pin, and selected GitHub release or tag pins used by downloaded artifacts and source checkouts. Runtime lanes are constrained by major version where the playbooks intentionally track a specific major, such as Node 22 for the default runtime and Node 24 for the Ansible language server runtime. Piper's Python runtime is intentionally not auto-updated because `python_versions` controls which pyenv runtimes Ansible installs; only change that runtime manually with the matching `python_versions` entry.

Custom managers for downloaded release assets should define `validationUrlTemplates` in `renovate.json` whenever the installed URL depends on the proposed version. `dotfiles-renovate-apply` checks those URLs before applying a candidate; HTTP failures such as `404` are reported as warnings and the candidate is skipped.

`dotfiles-renovate-apply` intentionally works from a temporary worktree so the checked-out branch in `~/dotfiles` is not modified. If updates are found, the worktree is left in `/tmp` with a committed local branch that can be inspected with Git, lazygit, or Diffview before merging or cherry-picking. If no updates are found, the empty worktree and branch are removed by default.

Unpinned pip tasks are outside Renovate pin proposals, but not outside update-diff capture. For pip, `update-diff-capture` records installed versions before and after the real Ansible pip task and captures an artifact diff when the observed version changes. Without an explicit target version there is no pre-run target for Renovate to propose, and `state: present` will not normally upgrade an already-installed package; use `state: latest` only where that moving behavior is intentional.

AUR review remains separate. `aur-review` owns installed-state discovery, reviewed AUR baselines, upstream VCS diffs, and `[aur-local]` publication state; Renovate should not replace that workflow unless the package is moved into a tracked dotfiles pin whose source can be modeled directly.
