# Browser Resource Limits

Chromium-family desktop launchers are generated by `playbooks/roles/10-system-tools/tasks/170-setup-browser-tools.archlinux.yml`. Keep browser resource controls there instead of editing generated files under `~/.local/share/applications` by hand.

The browser launchers call `~/bin/app-cgroup-launch`, which starts Brave and Chromium in transient user `systemd` scopes. This keeps browser subprocesses in one cgroup and allows limits to be adjusted from Ansible defaults:

```text
playbooks/roles/10-system-tools/defaults/main/app-resources.vars.yml
```

Use `MemoryHigh` as the first tool for memory pressure. It throttles/reclaims without immediately killing the browser. Keep `MemoryMax` as a higher hard safety rail, because reaching it can terminate the whole browser scope. For CPU, prefer `CPUQuota` when a hard cap is wanted and `CPUWeight`/`Nice` when the browser should yield to other interactive work.

Only browser processes started through the generated `.desktop` entries inherit these limits. If a browser was already running from an uncapped launch, a later launcher invocation may only hand URLs to that existing instance; restart the browser from the generated launcher before judging the cgroup limits.

Other desktop apps that need the same transient-scope pattern should also use `~/bin/app-cgroup-launch`, generated by `playbooks/roles/10-system-tools/tasks/165-setup-resource-limits.archlinux.yml`.

As with browsers, desktop-file overrides only affect new app launches. Quit an already-running single-instance app before judging the new cgroup limits.
